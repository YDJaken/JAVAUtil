# 要求和设计目标一览:
做一个single page web app, 里面有一个搜索框，用户输入内容之后可以去github上面搜索关键字。

可以随着用户的输入显示返回的结果。

样式不限，使用的库不限。

使用Github的GraphQL API v4

#要求
```
1.只有在用户的键盘停止输入的时候才会发起搜索请求。
2.可以显示历史记录
3.可以比较搜索结果
4.可以排序或者按条件过滤
```

## 需求分解
```
场景:
用户使用APP内搜索功能从github按搜索关键词获取结果,并对返回的结果可以查看历史记录,比较和进行排序筛选

此场景下用户可能的痛点:
    1.尽快得到结果
    2.方便的对比和回溯

此场景下可能的问题:
    1.终端机性能可能有限(内存使用有限制)
    2.网络可能不稳定
    3.终端机可能分辨率不固定
    4.最终场景交付可能需要模块化的嵌入页面

针对痛点和问题的设计框架:
    1.更快: 
        i.降低请求开销、减少请求时间 默认创建5个XMLHttpRequest对象用于直接开启请求，减小开销,如果要求更高可以改进至woker请求,如果仍不能满足则可以增加后端redis缓存+数据库存储
        ii.增加同时请求数 默认和XMLHttpRequest的可用请求数相同
        iii. 限制数目 默认请求关键字匹配前15个
    2.将用户请求过的历史缓存于页面内存中, 默认缓存XMLHttpRequest对象数*3
    3.通过调整前几个参数保证内存，提供请求历史删除机制和淘汰机制
    4.增加重试机制,重试3次后算作请求失败,告知用户
    5.模块使用百分比+最小height,width配合布局 (很多其他细节并没有在此模块考虑,比如全局字号，字体样式，各种dom类型的标准样式，输入框的长短等)
    6.允许渲染入口函数传入domID,保证其他内部渲染在此dom节点下，最差情况只会把此dom元素撑开至minWidth,minHeight

删除机制细节:
    1.由用户在页面点击删除

淘汰机制细节:
    1.根据用户兴趣度排序
    2.用户兴趣度排序方式:
       i.初始以时间排序(最新搜索的最前)
      ii.在用户点击回顾历史和点击对比后将对应历史提高至最高兴趣度
    3.在队列满员后将最低兴趣度删除
```

## 测试ApolloClient请求发送机制
```
1.ApolloClient暂时不支持限制同时请求个数
2.ApolloClient的请求每次均是新创建XMLHttpRequest
3.预创建XMLHttpRequest并限制同时请求个数对服务器和终端系统均更加友好
4.测试代码在src/test.js 
5.综合考虑其他需求后决定使用自行封装的创建XMLHttpRequest
```
